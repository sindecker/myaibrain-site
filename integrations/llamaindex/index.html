AIBrain + LlamaIndex
Persistent Memory for LlamaIndex Agents

AIBrain gives LlamaIndex chat engines and agents a persistent memory layer that survives session boundaries. Without AIBrain, every LlamaIndex session resets. With AIBrain, relevant memories from past sessions are injected as context on every new query — your agent remembers what matters.

$ pip install aibrain

Quick Start

Use the from_defaults factory — matches LlamaIndex naming conventions. Pass it to any SimpleChatEngine or agent.

from aibrain.integrations.llamaindex_adapter import AIBrainChatMemory
from llama_index.core.chat_engine import SimpleChatEngine

# Use the from_defaults factory (LlamaIndex convention)
memory = AIBrainChatMemory.from_defaults(user_id="llamaindex-alice", k=5)

engine = SimpleChatEngine.from_defaults(memory=memory)
engine.chat("I work at Acme Corp in the AI team.")

# Next session, same user_id:
engine.chat("Where do I work?")
# → "You mentioned you work at Acme Corp in the AI team."

Why AIBrain for LlamaIndex

Without AIBrain, LlamaIndex chat engines reset on every session. Context from prior conversations is gone. AIBrain adds a LlamaIndex memory layer that retrieves semantically relevant memories from past sessions and injects them as a SYSTEM message before each response — your agent remembers without bloating the context window.

Full BaseMemory Interface

Implements get, put, get_all, set, reset, to_string, and from_string. Works anywhere LlamaIndex expects a memory object.

Semantic Retrieval, Not Full Replay

AIBrain searches for the k most relevant prior memories for each input. Context windows stay lean — no raw history dumps.

In-Session + Cross-Session

In-session chat is buffered for instant access. Memories persist to AIBrain for long-term recall across deployments and machine reboots.

Multi-User, Multi-Agent Scoping

user_id and agent_id scoping supports safe multi-user, multi-agent deployments without memory bleed-over.

from_defaults Factory

Follows LlamaIndex naming conventions — slots into existing codebases without friction and no migration needed.

Zero Infrastructure

No Redis, no Postgres. Local SQLite. Works on any machine, works offline. No cloud dependency.
